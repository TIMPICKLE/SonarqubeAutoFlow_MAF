# MAF 迁移实现说明与文档勘误

## 一、迁移实现总结

本项目将 SonarQube 代码异味自动修复系统从 LangGraph 迁移到 Microsoft Agent Framework (MAF) 1.10.0。

### 迁移完成项

| 模块 | 迁移方式 | 状态 |
|------|---------|------|
| ReAct 循环 (`_react_loop` 200+ 行) | MAF Agent 内置 tool loop | ✅ 已删除，由 `Agent(client, instructions, tools)` 替代 |
| GLM API (`APIs/glm_5_api.py`) | MAF `OpenAIChatClient` | ✅ OpenAI 兼容接口直接接入 |
| MCPClient (`main.py:81-307`) | 轻量 `MCPManager` (~150 行) | ✅ 去除手动 event loop/thread，保留同步调用能力 |
| ToolRegistry + BaseTool | MAF `FunctionTool` (`@tool` 装饰器) | ✅ 声明式注册，去除两阶段选择 |
| LangGraph StateGraph | Python 线性编排 | ✅ 保留相同的工作流逻辑 |
| WorkflowState (TypedDict) | `WorkflowState` 类 | ✅ 作用域隔离 |
| 密钥硬编码 | 环境变量 + `.env` | ✅ 已环境变量化 |

### 复用的业务逻辑（与框架无关，直接复用）

- `claude_code_fix`：Claude Code CLI 调用逻辑（prompt 构建、git 提交推送、日志记录）
- `analyze_smell_type` / `analyze_complexity`：异味分类与复杂度评估
- `read_source_code` / `read_full_file` / `search_similar_patterns`：代码读取
- Issue 分析、工作区设置、PR 创建、记录保存、失败记录：业务逻辑全部保留
- 飞书通知（群组 webhook + 私信 lark-oapi）

## 二、对原始对比文档 (MAF_vs_LangGraph_对比分析.md) 的勘误

在基于真实 MAF 1.10.0 源码（PyPI 包 + GitHub 仓库 + 已安装包内省）核实后，发现原文档有以下需修正之处：

### 勘误 1：MCP API 描述错误

**原文档写道**：
```python
from agent_framework.tools import MCPToolset, SseServerParams
sonarqube_tools = MCPToolset(params=SseServerParams(url="..."))
```

**实际情况**：`agent_framework.tools` 模块不存在，`MCPToolset` 和 `SseServerParams` 类也不存在。

**真实 API**：
```python
from agent_framework import MCPStdioTool, MCPStreamableHTTPTool, MCPWebsocketTool

# stdio 模式（本项目使用的 npx 启动方式）
tool = MCPStdioTool(name="sonarqube", command="npx", args=["--yes", "sonarqube-mcp-server@latest"], env={...})

# HTTP 模式
tool = MCPStreamableHTTPTool(name="...", url="...")

# 或通过 chat client
tool = client.get_mcp_tool(name="...", url="...", headers={...})
```

### 勘误 2："Custom Provider" 说法误导

**原文档写道**：需要搭建 "GLM-5.2 Custom Provider"。

**实际情况**：GLM 提供 OpenAI 兼容的 `/v1/chat/completions` 接口，直接使用 MAF 的 `OpenAIChatClient` 即可，无需编写任何 Custom Provider：

```python
from agent_framework.openai import OpenAIChatClient
client = OpenAIChatClient(model="glm-5", api_key="...", base_url="https://ai-infra.united-imaging.com")
```

### 勘误 3：附录 C 示例代码不可运行

原文档附录 C 的 `CustomChatClient`、`MCPToolset(params=SseServerParams(...))` 均为臆造 API，无法直接运行。

### 勘误 4：版本信息

原文档列出的版本号基本准确：Python `agent-framework` 1.10.0 已确认存在于 PyPI（截至 2026-07）。

## 三、迁移中保留的设计决策

### 为何未使用 MAF Workflow (WorkflowBuilder)

MAF 提供 `WorkflowBuilder` / `Executor` / `Workflow` 用于图式工作流编排。但本项目的原始工作流是简单的线性 DAG（issue_analysis → workspace_setup → agent_fix → pr_creation → record_keeping），条件路由仅基于 error_info。

使用 MAF WorkflowBuilder 会引入额外复杂度（Executor 定义、Edge 配置、状态序列化），而线性编排 + error 路由用 Python 直接实现更简洁清晰，且行为与原项目完全一致。MAF 的核心价值（Agent + Tool loop）已在 SmellFixAgent 中充分体现。

### 为何保留轻量 MCPManager 而非全用 MCPStdioTool

MAF 的 `MCPStdioTool` 是设计为 Agent 工具（通过 function calling 调用）。但本项目的 `issue_analysis` 和 `pr_creation` 是确定性节点（查询 issues、创建 PR），不需要 LLM 决策，直接编程式调用 MCP 更合适。因此保留了一个极简的同步 MCPManager（~150 行），相比原始 MCPClient（200+ 行）去除了手动 event loop / thread 管理。

### 为何去除 ToolRegistry 两阶段选择

原始 `ToolRegistry` 的两阶段选择（规则筛选 + LLM 决策）是为 GLM-5 不支持 function calling 而设计的——通过缩小工具集降低 LLM 选择难度。MAF 的 Agent + function calling 由框架自动处理工具选择，无需手动缩小候选集。因此所有工具直接声明式注册到 Agent，由 LLM 原生 function calling 选择。
